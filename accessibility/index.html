Accessibility

Xooie relies on semantic relationships between elements. When possible, Xooie roles are derived from ARIA roles. Even if your markup isn't 100% semantic, Xooie will add the appropriate attributes to make sure that the contextual relationships between elements are defined. As a result, Xooie widgets are highly compatible with screen readers and other assistive technologies.

How Xooie Makes Accessibility Easier

Determining which attributes to add to make a widget accessible can be inordinately complicated:


        <div>
            <ul role="tablist" aria-owns="tab-1 tab-2 tab-3"></ul>

            <button id="tab-1" role="tab" aria-controls="tabpanel-1" aria-selected="true">Header 1</button>
            <div id="tabpanel-1" role="tabpanel" aria-expanded="true">Content for header 1</div>

            <button id="tab-2" role="tab" aria-controls="tabpanel-2" aria-selected="false">Header 2</button>
            <div id="tabpanel-2" role="tabpanel" aria-expanded="false">Content for header 2</div>

            <button id="tab-3" role="tab" aria-controls="tabpanel-3" aria-selected="false">Header 3</button>
            <div id="tabpanel-3" role="tabpanel" aria-expanded="false">Content for header 3</div>
        </div>

Xooie takes the complexity out of building accessible widgets by just defining roles. When adding certain elements is non-intuitive, Xooie will render those elements for you:


        <div data-widget-type="tab">

            <button data-x-role="tab">Header 1</button>
            <div data-x-role="tabpanel">Content for header 1</div>

            <button data-x-role="tab">Header 2</button>
            <div data-x-role="tabpanel">Content for header 2</div>

            <button data-x-role="tab">Header 3</button>
            <div data-x-role="tabpanel">Content for header 3</div>
            
        </div>

Nothing beats good old semantic markup, but Xooie takes the complication out of building rich, accessible widgets for the web
